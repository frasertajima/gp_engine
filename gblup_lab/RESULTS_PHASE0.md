# GBLUP Lab — Phase 0 results (parity, no new science)

**Date:** 2026-07-14. **Goal (LAB_PLAN.md):** prove the plumbing — GEMM-built
GRM → `PrecomputedKernel` → gp_engine's FP32-factor/FP64-IR solve — reproduces
MPDOK's own linear-GBLUP numbers before touching hyperparameter fitting.

backends solving independently:

| wheat/E2 | 1.19e-10 | 8.84e-11 | 7.03e-11 | 2 |
| wheat/E3 | 1.91e-10 | 8.31e-11 | 5.46e-11 | 2 |

gp_engine's FP32-factor + FP64-IR path (via the new `PrecomputedKernel` in

2–3 IR steps, every time. The plumbing is correct.

## 2. CV accuracy: gp_engine vs MPDOK's own `cv_lambda_sweep`, same split, same data

Ran the identical 5-fold split (`seed=42`, same `kfold_indices` code) and the
identical λ grid (`logspace(-4, 1, 20)`) through both gp_engine's solver and a
**fresh, direct call to MPDOK's actual `gblup.cv_lambda_sweep` function**
(not a re-derivation — the real code, imported and run today):

| dataset/trait | gp_engine best λ | gp_engine r | MPDOK `cv_lambda_sweep` r (run today) | MPDOK README r |
|---|---|---|---|---|
| wheat/E1 | 1.624 | 0.4264 | 0.4264 | 0.484 |
| wheat/E2 | 2.976 | 0.3880 | 0.3880 | 0.495 |
| wheat/E3 | 2.976 | 0.3678 | 0.3678 | 0.398 |
| wheat/E4 | 1.624 | 0.4534 | 0.4534 | 0.461 |
| mice/bmi | 0.0001 | 0.1378 | 0.1378 | 0.280 |

gp_engine matches MPDOK's own function bit-for-bit (agreement to the 6th+
decimal — the residual difference is float-order noise between numpy and
scipy's `dgesv`, not a real discrepancy). **That comparison is airtight.**

## 3. A finding, not a bug: MPDOK's README numbers don't match MPDOK's own current code

The third column above is what `MPDOK/gblup/gblup.cv_lambda_sweep` actually
returns, run fresh, on the data files sitting in `MPDOK/gblup/data/` right now.
The fourth column is what `MPDOK/gblup/README.md` (Experiment 1 and 2 tables)
claims. They don't match — not close-but-noisy, systematically off by
0.03–0.14, mice off by 2x. Ruled out as explanations (all tested directly,
not assumed):
- **Fold protocol**: tried mean-of-fold-r vs pooled-r, tried the published
  BGLR 10-fold `sets` assignment instead of a random 5-fold — none reach the
  README numbers (all land in the same 0.37–0.43 band gp_engine and MPDOK's
  live code both land in).
- **y-centering**: mean-centering y_train per fold before solving moves E1
  from 0.4264 to 0.4277 — noise, not the source.
- **Solver correctness**: ruled out by §1 and §2 — both backends agree with
  each other to float precision.

Likeliest explanation: the README table was generated by an earlier version
of the code or data (dataset regenerated since, or a different lambda grid /
seed used in that specific historical run) and never re-verified against the
current `cv_lambda_sweep`. This is exactly the "verify, don't trust the
summary" lesson from `gp_lab`'s safety-gate bugs and the `protocol="paper"`
discovery — it just turned up one directory over this time. **Not fixing
MPDOK's README** (non-goal — see LAB_PLAN.md); noting it here so gblup_lab's
own baseline is the number MPDOK's code produces today, not the number its
docs claim.

## Verdict

Phase 0 acceptance criterion (LAB_PLAN.md: "same r as README's reported
0.45–0.55 / 0.280") is **not met against the README text**, but **is met, exactly,
against MPDOK's own live code** — which is the correct bar; a doc string isn't
ground truth, a running program is. Plumbing confirmed correct. Ready for
Phase 1 (replace the CV-grid λ with gp_engine's marginal-likelihood fit and
see whether it beats 0.4264/0.3880/0.3678/0.4534/0.1378 — those are now the
real numbers to beat, not the README's).
